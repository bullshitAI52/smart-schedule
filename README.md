# 智能排班系统 - 算法规则与维护文档

本系统基于 `Vue 3` 构建，核心完全封装于 `index.html` 的 `autoSchedule`（一键智能排班）函数中。为了方便未来接手的开发者或您自己修改排班规则，以下记录了目前系统运转的**核心参数**与**硬核拦截算法**的工作原理。

## 1. 核心铁律（基础变量）
在 `autoSchedule` 函数开头，有几个写死的核心参数，可以直接修改它们来适应未来的基础变动：
* `totalEmps = 10`：当前设定的总员工数为10人。
* `minWorkers = 8`：每天**至少且必须保证** 8 名安保在岗。
* `maxRestDays` (可调)：每个员工在一个月内会被**精确凑满**指定的休息额度（默认为 4 天）。如果手动设置的休息天数超过此数值，统计栏将显示红色警告。

## 2. 班次定轨逻辑 
不再全盘随机分配 A 和 B。目前的规则是：
* **白班组（A）：** 永远由排在表格**前 5 行**（下标 0 到 4 也就是成员1~5）承包。
* **晚班组（B）：** 永远由第 6 行及以后的人承包。
* *如需修改：在源码最底部的 `emp.schedule[d] = index < 5 ? 'A' : 'B';` 中调整 `5` 这个控制线即可。*

## 3. 防单班次薄弱：单班组同日限休 1 人
为了绝对保障某天的白班或者晚班不会出现“2人同休导致岗位空虚”的状况。
算法在尝试向某一天插入“休”字时，会实时计数：
* 如果尝试放给白班（A）人员，系统会检查当天的白班组是不是已经有 1 个休假了 （`aRests >= 1`）。如果是，立刻驳回并换一天。晚班（B）同理。

## 4. 超进化算力：间隔拉扯算法
这是保障员工状态“不会太累、全月分布均匀”的最高级约束模块。系统在排某个人的休假时，会自动提取他这一个月的休假快照，并向前后计算“纯工作日的间隔天数（Gap）”：
* **拦截连休 3 天：** 只要发现放入“休”会导致形成“休、休、休”，一概驳回 (`spacingValid = false`)。
* **单休间距大：** 如果放入的是孤立的单休（比如 `...上、休、上...`），则强制它和上下一次的单休之间**至少空出 5 天纯工作日**（`gap < 5` 则驳回）。这保证了单休是一周休1次左右（间隔 6~7 天）。
* **连休 2 天的超大隔离带：** 如果出现了极少见的 `休、休` 两天连放，算法会强制它左侧的上班天数或右侧的上班天数**绝不可小于 8 天** （`leftGap < 8` 或 `rightGap < 8` 则驳回）。这就实现了“既然连休2天，就大概隔10天的大跨度”。

## 5. “尽量每天都有人休”的 50 倍轮盘抽签算法
为了避免把休假额度堆积在月末或者少部分天数上，排班抽签在选取日子的时候，采取了“加权冷门日子”策略：
* 当某一天**目前 0 个人休息（为空）** 时，系统会给这一天的权重赋予 `50` 的极高倍率 （`weight = 50`）；
* 只有当绝大多数日子都有了 1 个人休息后，后续名额才会开始砸向 `weight = 1` 的日子。
这几乎从数学概率上保证了一张表出来，每天都最少有 1 人休息。

## 6. 万次纠错重启机制 (MAX_RETRIES)
由于上述 5 条规则异常苛刻且层层嵌套。如果纯人工排班很可能会排进死胡同导致解不开。
这套系统采用的是类 **Las Vegas 退火算法**：
* 每次给员工安排休假前，会自动备份现有的“受保护的手动设置数据”。一旦上面的极高要求导致在日历走入死胡同排不下去。
* 算法会在静默的 `0.05` 秒内，抛弃全盘的脏数据重新抽取组合。
* `MAX_RETRIES = 10000`：即系统被允许试错并自己打翻重来 1 万次。如果 10000 次都走不通，才会弹窗提示管理人员“因为极限冲突，目前排法无解”，从而极大地保障程序的健壮性。

--- 
**未来需求修改提示录：**
1. **想改月休天数？** 直接在页面顶部的“月休限制”输入框中修改即可。
2. **想改连休或单休间隔？** 搜索代码里的 `leftGap < 5` 以及 `< 8` 并替换为您所需的数字。
3. **增加总人员？** 默认名额 `const employees = ref(Array.from({length: 10}` 里的 `10` 改大即可支持扩编。
4. **红色警告？** 如果您手动给某人排了超过设定的月休天数，系统会在“休”这一列标红提醒。
